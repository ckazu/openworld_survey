# ハイポリ・プロシージャル化 設計

- **日付**: 2026-06-11
- **対象**: Three.js オープンワールドデモ（草原・湖畔・森）
- **目的**: ローポリ表現をハイポリ・リアル方向へ底上げする
- **方針**: 「外部アセットに一切依存しない完全プロシージャル生成」という現アーキの哲学を維持したまま、メッシュ密度・実行時生成 PBR テクスチャ・スクリーンスペース AO でリアルさを引き上げる（アプローチ A）

## スコープ

主対象は以下の 3 要素（ブレインストームでユーザーが選択）：

1. **木・岩の形状** — ブロブからの脱却、枝構造・葉クラスタ・岩肌ディテール
2. **草の密度・質感** — 形状バリエーション、サブサーフェス透過、密度向上
3. **光と陰影** — スクリーンスペース AO の追加とライティング再バランス

性能予算は「高性能 GPU・見た目最優先」。FPS より見た目を優先する。

### 非スコープ

- 地形の質感（ディスプレイス/PBR テクスチャ）— 今回は対象外
- 外部 GLTF モデル・外部画像テクスチャの導入 — 哲学に反するため不採用
- 水面の屈折・SSR — 今回は対象外（既存の反射を維持）
- 時間変化・天候などの新機能 — YAGNI

## アーキテクチャ

既存のモジュール構成（`terrain.js` / `water.js` / `sky.js` / `vegetation.js` /
`grass.js` / `ambience.js` / `player.js` / `noise.js` / `main.js`）を維持し、
新規ユーティリティ 1 ファイルを追加する。

### 新規: `src/textures.js`

ノイズから実行時に PBR マップ（`THREE.DataTexture`）を生成するユーティリティ。
外部画像ゼロの哲学を守る中核。

- `generateNormalMap(size, heightFn, strength)` — 高さ場 `heightFn(x, y)` を
  Sobel 演算子で微分し、接空間法線（RGB）を `DataTexture` に書き出す
- `generateRoughnessMap(size, fn)` — FBM 由来のラフネス（グレースケール）
- 既存 `noise.js` のタイル化可能 FBM を高さ場に用い、テクスチャを継ぎ目なく
  タイリング可能にする（`wrapS/wrapT = RepeatWrapping`）
- 木の樹皮・岩肌などプリセットを関数として公開し、各モジュールから呼ぶ
- 生成は初期化時に一度だけ。テクスチャは material 間で共有しメモリを節約

**インターフェース**: 入力＝サイズと高さ関数、出力＝`THREE.Texture`。
呼び出し側は内部実装を知らずに `material.normalMap` に挿すだけでよい。

## コンポーネント設計

### 1. 木・岩（`vegetation.js` 改修）

既存の `scatter` / `buildInstances` / `InstancedMesh` 構造・`castShadow` は
そのまま流用（インスタンシングによる性能を維持）。ジオメトリ生成だけ刷新する。

#### 共通ヘルパ追加

- `createBranch(...)` — テーパー円柱を曲げ・分岐させて枝を生成
- `createLeafCluster(...)` — 反りのある小平面（葉カード）を多数、指定ボリューム内に
  ばら撒いて 1 ジオメトリにマージ。葉カードの法線は外向き＋上向き寄りに調整し、
  ライティングで塊として見えるようにする

#### 針葉樹

- 幹：テーパー円柱（既存）＋数本の短い枝。樹皮法線マップ適用
- 樹冠：ブロブ 3 段の代わりに、円錐状ボリュームへ葉カードクラスタを配置。
  下が暗く上が明るい頂点カラーは維持

#### 広葉樹

- 幹：テーパー円柱＋枝を数本（上方で分岐）。樹皮法線マップ適用
- 樹冠：枝先に葉カードクラスタを広げる。葉は軽量サブサーフェス
  （裏面ライティング項）で逆光時に透ける

#### 岩

- `IcosahedronGeometry` detail=3〜4 ＋ FBM 多重変位（既存 `displace` を多重化）で
  割れ目・段差を作る
- プロシージャル岩肌の法線マップ＋ラフネスマップを適用（PBR）
- 頂点カラーは露出岩肌と苔のブレンド（高さ/法線由来）

### 2. 草（`grass.js` 改修）

- ブレード形状を 3〜4 種に増やす（細い／幅広／反りの違い）。
  ジオメトリ配列を持ち、タイル内でインスタンスごとにランダム選択
  （= 形状ごとに InstancedMesh を分ける、もしくは頂点で形状を切替）
- サブサーフェス透過：`onBeforeCompile` のフラグメント側に、太陽方向と
  ビュー方向に基づく裏面透過項を追加し、逆光で穂先が透ける
- 近距離タイルの密度を `BLADES_NEAR` 1500 → 2200 程度へ（性能を見て調整）
- 穂先の色バリエーションを強化（既存 HSL パッチを拡張）
- 既存の 2 段 LOD・タイル動的生成・レイヤー 1 除外は維持

### 3. 光と陰影（`main.js` / `sky.js` 改修）

- `postprocessing` パイプラインに **GTAO パス**
  （`three/addons/postprocessing/GTAOPass.js`）を追加。
  挿入位置は `RenderPass` の直後、`UnrealBloomPass` の前
- GTAO の `output` は通常合成、半径・強度はシーンスケールに合わせて調整
- トーンマップ露出・影 `bias`・IBL 強度（`environmentIntensity`）を再バランスし、
  AO 追加後に暗くなりすぎないよう補正
- 既存 MSAA レンダーターゲット・Bloom・カラーグレードは維持

## データフロー

```
noise.js (FBM/タイル化FBM)
  └─> textures.js (DataTexture: normal/roughness)
        ├─> vegetation.js (樹皮/岩肌 material)
        └─> grass.js (任意)
main.js
  RenderPass -> GTAOPass -> UnrealBloomPass -> OutputPass -> ColorGradePass
```

## エラー処理・互換性

- `GTAOPass` は WebGL2 前提（既存も WebGL2 想定）。利用不可環境のフォールバックは
  今回スコープ外（性能予算が高性能 GPU 前提のため）
- テクスチャ生成は初期化時に同期実行。サイズは 256〜512 程度に抑え、
  生成コストとメモリを管理
- インスタンス数・ブレード密度は定数化し、性能調整を一点で行えるようにする

## テスト・検証

Three.js のリアルタイムデモのため自動ユニットテストは最小限とし、視覚検証を主とする。

- `npm run dev` で起動し、Playwright でスポーン地点と森・草原・岩付近の
  スクリーンショットを撮影、Before/After を比較
- 確認観点：木の樹冠がブロブでなく葉の塊に見えるか／岩肌のディテール／
  逆光で草が透けるか／接地の AO が効いているか／FPS が破綻していないか
- `window.__demo` デバッグフック（`freeze` 等）は維持し、検証に活用

## 実装順序

1. `textures.js` 基盤（法線/ラフネス生成）
2. 岩のハイポリ化（最短で効果が大きい）
3. 木のハイポリ化（針葉樹 → 広葉樹）
4. 草の形状バリエ・サブサーフェス・密度
5. GTAO 追加とライティング再バランス

各段でスクリーンショット確認・コミットを行う。
